This project uses Bun, TypeScript, oxlint, and Bun's test runner.

## Commands

```bash
bun run lint
bun run typecheck
bun test
bun run build
bun run check
bun run ci
bun run deploy:package
bun run deploy:site
```

`bun run ci` is the full monorepo quality gate. `bun run check` is the package-only quality gate:

1. `oxlint .`
2. `tsc --noEmit`
3. `bun test`
4. `bun run build`

Site-only checks live in the site package:

```bash
bun run --cwd site ci
```

## Build

The build intentionally avoids `tsup` and esbuild.

```bash
rm -rf dist
tsc -p tsconfig.build.json
bun scripts/fix-dist-imports.ts
```

TypeScript emits ESM and declarations. The Bun script rewrites local generated ESM imports to include `.js` extensions so `dist/index.js` works in Node-style ESM environments.

## Deploy

The repo has separate deploy commands because the npm package and docs site ship to different places:

```bash
bun run deploy:package
bun run deploy:site
bun run deploy:site:prod
```

- `deploy:package` publishes the built npm package.
- `deploy:site` creates a Vercel preview deployment for the docs site.
- `deploy:site:prod` deploys the docs site to production.

Run `bun run ci` before any deploy.

## Source Layout

Published source lives under `src/`. Tests live under `tests/`. The Next.js and Fumadocs site lives under `site/`.

See [Project Structure](./project-structure) for the full tree.

## Tests

Tests are focused on pure behavior:

- Relationship helper rows.
- Subject-centered family indexing.
- Explicit neighborhood limits.
- Measured layout derivation.
- Relationship graph traversal.
- Theme variable mapping.
- Primitive SSR markup for data attributes, accessibility props, custom card props, and viewport scale.

## Linting

Oxlint is the only linter. Keep `bun run lint` at zero warnings.

## Browser Verification

After layout or rendering changes:

1. Run the docs site with `bun run dev` from `site/`.
2. Open `http://localhost:3000`.
3. Inspect the console.
4. Capture a screenshot of the homepage family tree demo and a docs page.

## Documentation

Keep the public docs aligned when changing the API:

- `README.md`
- `AGENTS.md`
- `site/content/docs/index.mdx`
- `site/content/docs/family-tree.mdx`
- `site/content/docs/api.mdx`
- `site/content/docs/design.mdx`
- `site/content/docs/org-chart.mdx`

New examples should prefer the clean public names: `people`, `subject`, `card`, and `onPersonClick`. Keep `profiles`, `rootProfileId`, `renderProfileCard`, and `onSelectProfile` documented as compatibility aliases.
